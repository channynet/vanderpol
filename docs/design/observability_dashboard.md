# Experiment Observability And Read-Only Dashboard Design

## Purpose

This document is the implementation contract for adding observability and a
local read-only web dashboard to the Vanderpol experiment pipeline.

The immediate pain is not that the project lacks a web UI. The real problem is
that long-running steps do not consistently emit machine-readable progress,
heartbeat, metrics, provenance, or artifact events while they run. A dashboard
without those signals would be a thin shell with unreliable progress bars.

This design therefore starts with instrumentation first, then builds a small
read-only UI on top of the recorded state.

The document is intentionally detailed so work can resume after context
compaction without losing intent, quality gates, or completion criteria.

## Non-Goals

The MVP must not become a general workflow orchestrator.

- No multi-user support.
- No authentication.
- No external database beyond append-only files; SQLite may be considered later.
- No WebSocket requirement; polling is enough.
- No React or large frontend framework for the MVP.
- No mobile-specific UI.
- No unsafe process suspend/resume design.
- No write-heavy control panel in the first version.

## Guiding Principles

1. Instrumentation before UI.
2. Append-only logs as the source of truth.
3. Atomic current snapshots for cheap UI polling.
4. Read-only MVP; controls come after checkpoint-based stop/resume exists.
5. Run reproducibility is a first-class output, not a nice-to-have.
6. Every long step must expose honest sub-unit progress.
7. ETA is approximate and must be labeled as such.
8. A stalled or crashed run must be distinguishable from a running run.

## Scope

The initial implementation covers local single-user experiment observation for
the existing bundle runner and long-running analysis steps.

Primary run target examples:

- `outputs/runs/stage9_n100`
- `outputs/runs/stage9_n100_time2`

Primary bundle steps:

- `phase2_figures`
- `calibration_report`
- `selector_report`
- `decision_boundary`
- `bootstrap_ci`
- `selector_stability`
- `noise_ood_sweep`
- `fallback_threshold_sweep`

## Architecture Overview

The pipeline writes structured files inside each run directory:

```text
outputs/runs/<run_id>/
  run_manifest.json
  run_progress.json
  events.jsonl
  metrics.jsonl
  artifacts.jsonl
  current_progress.json
  failure_summary.json
  logs/
    ...
```

The dashboard reads those files and never mutates the run in the MVP.

Recommended local server:

- Bind address: `127.0.0.1`
- Default port: `8765`
- Backend: FastAPI or Flask
- Frontend: static HTML, CSS, and vanilla JavaScript
- Refresh: polling every 2-5 seconds

## Phase 0: Instrumentation Contract

Phase 0 is mandatory before building the dashboard.

### Required Files

#### `events.jsonl`

Append-only event stream. This is the historical source of truth.

Each line is one JSON object. Event writes should append, flush, and fsync where
practical. If fsync is too expensive for very frequent events, heartbeat events
may be throttled.

Required common fields:

```json
{
  "schema_version": 1,
  "event_id": "uuid-or-stable-id",
  "run_id": "stage9_n100_time2",
  "event_type": "step_progress",
  "created_at": "2026-05-30T02:12:00+00:00",
  "step": "fallback_threshold_sweep",
  "status": "running",
  "message": "Precomputed profile 6/10: real_asystole"
}
```

Supported `event_type` values:

- `run_started`
- `run_heartbeat`
- `run_done`
- `run_failed`
- `step_started`
- `step_progress`
- `step_done`
- `step_failed`
- `metric`
- `artifact_written`
- `warning`

#### `current_progress.json`

Atomic snapshot generated by folding the latest known run state. The UI should
read this file first because it is small and cheap.

Writes must use temp-file then rename:

```text
current_progress.json.tmp -> current_progress.json
```

If the UI catches a parse failure, it must ignore that poll and retry on the
next interval.

Required fields:

```json
{
  "schema_version": 1,
  "run_id": "stage9_n100_time2",
  "status": "running",
  "current_step": "fallback_threshold_sweep",
  "heartbeat_at": "2026-05-30T02:12:00+00:00",
  "started_at": "2026-05-30T01:30:00+00:00",
  "updated_at": "2026-05-30T02:12:00+00:00",
  "completed_steps": 7,
  "total_steps": 8,
  "progress_fraction": 0.875,
  "eta_seconds": 3600,
  "eta_method": "rough_linear_subunit_estimate",
  "steps": []
}
```

#### `metrics.jsonl`

Append-only scalar metrics for TensorBoard-lite charts.

Example:

```json
{
  "schema_version": 1,
  "run_id": "stage9_n100_time2",
  "created_at": "2026-05-30T02:12:00+00:00",
  "step": "noise_ood_sweep",
  "name": "selector_linucb.mean_reward",
  "value": 84.2,
  "x": 3,
  "x_name": "profile_index",
  "unit": "reward"
}
```

Initial metric candidates:

- `mean_reward`
- `oracle_gap`
- `success_rate`
- `mean_energy`
- `mean_time_s`
- `mean_safety_violations`
- `fallback_rate`
- `completed_profiles`
- `completed_configs`

Metric validation must detect NaN and inf. The first implementation may emit a
`warning` event and show a red banner; automatic stop is deferred.

#### `artifacts.jsonl`

Append-only artifact registry.

Example:

```json
{
  "schema_version": 1,
  "run_id": "stage9_n100_time2",
  "created_at": "2026-05-30T02:12:00+00:00",
  "step": "noise_ood_sweep",
  "path": "outputs/runs/stage9_n100_time2/noise_ood_sweep.csv",
  "kind": "csv",
  "status": "final",
  "size_bytes": 12345,
  "sha256": "optional-for-final-artifacts"
}
```

Artifact `status` values:

- `partial`
- `final`
- `stale`
- `failed`

Hashing policy:

- Final small/medium artifacts should get SHA-256.
- Very large or frequently changing partial artifacts may skip hashing and rely
  on size + mtime until finalized.

#### `failure_summary.json`

Only present after a failure or stall classification.

Required data:

- failed step
- exception class if available
- exception message
- traceback tail
- stderr tail
- last heartbeat
- last progress event

This is the file that powers the dashboard top failure banner.

## Provenance Requirements

Every run manifest must include enough data to reproduce and cite a result.

Required fields:

- `run_id`
- resolved config, not only the user override
- `reward_weights`
- all seeds
- `n_jobs`
- start/end timestamps
- hostname
- platform
- Python version
- package versions
- git commit if available
- git dirty state if available
- fallback text if the directory is not a git repository
- command line used to launch the run

Current repository note: this workspace may not be a git repository. The
implementation must handle that gracefully and record `"git": {"available":
false}` rather than failing.

## Unit Progress Definitions

Progress percentages must be based on real work units, not guessed display
states.

### `phase2_figures`

Unit definition:

- scenario-patient simulation block

Suggested counters:

- `completed_patients`
- `total_patients = patients_per_scenario * number_of_scenarios`
- `completed_rows`
- `total_rows = total_patients * number_of_algorithms`

### `calibration_report`

Same unit definition as `phase2_figures`, plus:

- `completed_targets`
- `total_targets`

### `selector_report`

Same algorithm matrix unit definition, plus:

- `train_contexts`
- `eval_contexts`

### `decision_boundary`

Units:

- algorithm matrix precompute patients
- grid cells after model training

Counters:

- `completed_grid_cells`
- `total_grid_cells = grid_size * grid_size`

### `bootstrap_ci`

Units:

- algorithm matrix precompute patients
- bootstrap draws

Counters:

- `completed_bootstrap_draws`
- `total_bootstrap_draws`

### `selector_stability`

Units:

- split seeds
- matrix patients per seed

Counters:

- `completed_seeds`
- `total_seeds`
- nested current seed patient progress when feasible

### `noise_ood_sweep`

Units:

- clean training matrix
- per-profile noisy matrix

Counters:

- `completed_profiles`
- `total_profiles`
- `current_profile`
- `completed_patients`
- `total_patients`

### `fallback_threshold_sweep`

Units:

- clean training matrix
- noisy matrix precompute per profile
- threshold config evaluation

Counters:

- `completed_profiles`
- `total_profiles`
- `current_profile`
- `completed_configs`
- `total_configs`
- `current_config`

This is the highest-priority long step to instrument because it already has
partial checkpointing and is the main source of uncertainty.

## Stalled, Crashed, Done Semantics

The UI must not blindly trust `"status": "running"`.

State classification:

- `completed`: terminal `run_done` event or all expected final outputs present.
- `failed`: terminal `run_failed` or `step_failed` event.
- `running`: heartbeat is fresh and no terminal event exists.
- `stalled`: status says running, but `heartbeat_at` is older than threshold.
- `unknown`: insufficient data.

Initial stalled threshold:

- Warning after 5 minutes without heartbeat.
- Strong stalled classification after 15 minutes without heartbeat.

The threshold should be configurable later.

## Notifications

Push notifications are valuable for unattended overnight runs, but they depend
on reliable events and heartbeat first.

Phase 1 notification targets:

- webhook URL for Slack, Discord, ntfy, or Telegram-compatible endpoint
- optional Windows toast later

Notify on:

- run completed
- run failed
- run stalled
- NaN/inf warning

The MVP must not require notifications to be configured.

## Read-Only Dashboard MVP

The MVP dashboard reads files only.

### Pages Or Panels

1. Run list
   - run id
   - status
   - current step
   - progress percent
   - last heartbeat
   - elapsed time

2. Run overview
   - resolved config summary
   - provenance summary
   - step progress
   - rough ETA
   - stale/stalled banner

3. Step timeline
   - eight bundle steps
   - status
   - duration
   - outputs count
   - latest message

4. Deep progress
   - current unit counters
   - profile/config/patient counts
   - current profile/config where available

5. Failure triage
   - visible only when failure/stall/warning exists
   - stderr tail
   - traceback tail
   - last event

6. Live metrics
   - simple line charts from `metrics.jsonl`
   - no TensorBoard dependency

7. Artifacts
   - table of artifacts
   - run display name from `outputs/runs/run_labels.json`
   - selected run storage folder
   - all run data files including manifest/progress/events/metrics/artifact logs
   - category grouping: paper results, figures, tables, reports, documents, logs, run metadata
   - important results are rendered in the dashboard without forcing the user to
     hunt through files:
     - paper or executive summary markdown
     - reward/success/time/safety heatmaps
     - decision boundary image
     - selector comparison table
     - scenario winner table
     - noise robustness table
     - fallback threshold table
   - path
   - type
   - partial/final status
   - size
   - updated time
   - preview links for markdown, png, csv, json, html

### MVP API

Suggested endpoints:

- `GET /api/runs`
- `GET /api/runs/{run_id}/progress`
- `GET /api/runs/{run_id}/events?tail=200`
- `GET /api/runs/{run_id}/metrics`
- `GET /api/runs/{run_id}/artifacts`
- `GET /api/runs/{run_id}/failure`
- `GET /api/runs/{run_id}/storage`

Server must bind to `127.0.0.1`, not `0.0.0.0`.

## Deferred Control Features

Do not implement these in the first read-only UI:

- pause
- resume
- stop
- delete run
- edit running parameters
- start new run from UI

When controls are added, use checkpoint-based graceful stop/resume.

Rejected design:

- Windows process suspend/resume using non-standard APIs

Accepted future design:

- create a stop request file
- long-running loops check it between work units
- current unit finishes
- checkpoint and terminal event are written
- process exits cleanly
- resume restarts the process and skips completed units

## Settings And Run Creation

Settings UI is deferred until read-only observation is stable.

When implemented:

- running run parameters are locked
- editing parameters creates a new run
- resolved config is written before execution
- dry-run cost estimate is shown before launch

Dry-run estimate MVP:

- total units
- rough unit count by step
- past-duration estimate only after enough events exist

## Quality Assurance Plan

### Unit Tests

Required tests:

- atomic snapshot write produces valid JSON.
- JSONL event append produces one parseable JSON object per line.
- event folding reconstructs current progress.
- stale heartbeat classification works.
- terminal done classification works.
- terminal failed classification works.
- artifact manifest records size and final/partial status.
- NaN/inf metric produces warning event.

### Integration Tests

Required tests:

- smoke bundle writes events and current progress.
- `fallback_threshold_sweep` writes profile/config progress.
- dashboard API reads a run with no failure.
- dashboard API reads a run with synthetic failure summary.
- dashboard handles malformed snapshot by falling back or retrying.

### Manual Verification

Minimum manual checks:

```powershell
$env:PYTHONPATH='src'
python -m unittest discover -s tests
python scripts/run_experiment_bundle.py --config configs/bundle_smoke.json --output-dir outputs/runs --run-id dashboard_smoke
python scripts/show_run_progress.py outputs/runs/dashboard_smoke
```

Dashboard verification:

- open `http://127.0.0.1:8765`
- run list shows `dashboard_smoke`
- current progress loads
- artifacts load
- no console JSON parse errors during polling

## UI Theme Attribution

The local dashboard visual shell uses the open-source TailAdmin/Tailwind Admin
template as a reference for layout and styling:

- Source: `Tailwind-Admin/free-tailwind-admin-dashboard-template`
- Local copy: `vendor/tailwind-admin-template`
- License: MIT, copyright `2026 Tailwindadmin`

We intentionally adapt the visual structure rather than introducing the
template build system. The dashboard remains a single locally served HTML page
from `src/vanderpol/dashboard.py`, with TailAdmin-inspired sidebar navigation,
top header, card surfaces, status badges, and artifact panels.

## Definition Of Done

This work is done when:

1. Long runs expose truthful progress without the dashboard.
2. A user can inspect progress by reading files directly.
3. The dashboard shows the same information without requiring log spelunking.
4. A failed run shows the failed step and useful traceback/stderr context.
5. A stalled run is distinguishable from an active run.
6. Result artifacts are registered with status and provenance.
7. All MVP tests pass.
8. Existing experiment outputs and scripts remain backwards-compatible.
